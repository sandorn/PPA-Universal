## 3. 功能扩展建议

### 3.1 高优先级（建议近期实现）

#### 3.1.1 形状快速复制与阵列

**需求背景**：快速复制形状并按规则排列（如 3x3 网格、环形阵列）。

**功能设计**：

- **矩阵复制**：指定行列数，自动生成网格排列
  - 输入：行数、列数、行间距、列间距
  - 输出：按网格排列的复制形状
- **线性复制**：指定数量和间距，水平或垂直复制
  - 输入：数量、间距、方向（水平/垂直）
  - 输出：线性排列的复制形状
- **环形阵列**：围绕中心点环形排列（可选，优先级较低）
  - 输入：数量、半径、起始角度
  - 输出：环形排列的复制形状

**实现建议**：

- 可扩展 `IShapeBatchService` 或新建 `IShapeDuplicateService`
- 使用 `IShapeOperations.CopyShape()` 进行复制
- 计算位置时考虑原始形状的边界框

**Ribbon 设计**：

```xml
<group id="grpDuplicate" label="复制">
  <button id="btnMatrixCopy" label="矩阵复制"
          imageMso="TableInsertGallery" size="large"
          onAction="OnMatrixCopy"/>
  <button id="btnLinearCopy" label="线性复制"
          imageMso="ObjectDuplicate" size="normal"
          onAction="OnLinearCopy"/>
</group>
```

#### 3.1.3 对齐基准切换

> **现状**：`PPARibbon.xml` 使用 `labelControl`「对齐参考」+ `dropDown`（`ddAlignRef`），回调 `OnAlignRefChanged` / `GetAlignRefIndex`，与 `IAlignmentService.Align(..., reference)` 联动。下文 **ToggleButton** 方案为可选替代设计，非当前 XML。

**需求背景**：对齐操作需要灵活切换参考基准，默认以形状为基准，有时需要以页面为基准。

**功能设计**：

- **基准切换按钮**：ToggleButton，默认状态为"形状基准"（未按下）
- **按下状态**：切换为"页面基准"（按下）
- **动态状态**：选中单个形状时，自动切换为"页面基准"
- **状态持久化**：记住用户选择，下次打开时保持上次状态

**实现建议**：

- 在 `RibbonCallbacks.cs` 中实现 `GetTbState` 回调，返回当前基准状态
- 实现 `TbOnAction` 回调，切换基准模式
- 在 `IAlignmentService` 中支持基准参数传递
- 使用配置或内存变量存储当前基准状态

**Ribbon 设计**：

```xml
<toggleButton id="Tb101" onAction="TbOnAction" size="large"
              screentip="对齐基准,默认形状为基准，按下或选中一个形状，则以页面为基准"
              getPressed="GetTbState" getLabel="GetTbLabel" getImage="GetIcon" />
```

#### 3.1.4 形状吸附功能

> **现状**：已实现 `IAlignmentService.SnapToShape`；Ribbon `supertip` 约定为 **先选对象为基准**，其余对象边与基准边**贴合共线**（与「对齐到同一边」不同）。

**需求背景**：在版式微调时，将对象沿某方向贴靠到基准对象的指定侧，形成边对边接触。

**功能设计**（与当前实现语义一致）：

- **向左吸附**：其余形状整体置于基准左侧，使**他形右缘 = 基准左缘**
- **向右吸附**：**他形左缘 = 基准右缘**
- **向上吸附**：**他形下缘 = 基准上缘**
- **向下吸附**：**他形上缘 = 基准下缘**

**实现建议**：

- 扩展 `IAlignmentService` 接口，添加 `SnapToShape()` 方法
- 获取第一个选中形状的边界作为参考
- 将其他选中形状对齐到参考形状的对应边缘
- 使用 `CreateUndoScope` 包裹操作以支持撤销

**Ribbon 设计**：

```xml
</box>
```

#### 3.2.2 颜色拾取与应用

> **文档状态**：以下为**需求草案**。当前仓库 **已移除** `IColorPickerService` / `ColorPickerService` 及 Ribbon 回调；落地时需重新建模（含 WPS 颜色属性差异）。

**需求背景**：快速拾取幻灯片中某个元素的颜色并应用到其他元素，提高配色效率。

**功能设计**：

- **颜色拾取器**：从选中形状/图片拾取填充色、边框色、文字色
- **颜色历史记录**：保存最近使用的 8-10 种颜色，支持快速选择
- **快速应用**：一键应用到选中形状的填充/边框/文字
- **颜色格式转换**：支持 RGB、HSL、主题色索引之间的转换和显示

**实现建议**：

- 新建 `IColorPickerService` 接口（当前代码库无此文件，需从零添加）
- 使用 `IShapeContext.Fill`、`IShapeContext.Line` 等属性获取颜色
- 颜色历史记录可存储在 `PPAConfig` 或用户配置文件中
- 提供颜色选择对话框（可选，或使用 Office 内置颜色选择器）

**技术要点**：

- PowerPoint: 通过 `Shape.Fill.ForeColor`、`Shape.Line.ForeColor` 等获取
- WPS: 需验证颜色属性访问方式是否一致

#### 3.2.3 形状锁定与解锁

**需求背景**：防止误操作移动或修改重要形状，特别适用于模板和固定布局。

**功能设计**：

- **锁定选中形状**：禁止移动、调整大小、删除（可选）
- **批量解锁**：解锁当前幻灯片/全部幻灯片的所有锁定形状
- **锁定状态可视化**：在形状上显示锁定图标或边框提示（可选）
- **锁定类型**：支持仅锁定位置、仅锁定大小、完全锁定

**实现建议**：

- 新建 `IShapeLockService` 接口
- 使用 `IShapeContext.Locked` 属性（如果平台支持）
- 如果不支持原生锁定，可通过标记或元数据实现
- 提供锁定形状列表视图（可选）

**技术要点**：

- PowerPoint: `Shape.LockAspectRatio`、`Shape.Locked` 属性
- WPS: 需验证是否支持相同的锁定 API

**功能设计**：

- **显示所有对象**：当没有选中对象时，显示当前幻灯片中所有隐藏的对象
- **状态切换**：支持隐藏/显示的切换操作

- 使用 `CreateUndoScope` 包裹操作以支持撤销

**Ribbon 设计**：

#### 3.2.5 边角料裁除

### 5.2 撤销/重做支持

**状态**：✅ 已实现

**工作原理**：

PowerPoint 的 COM 自动化行为：连续的 COM 操作会被自动合并为一个撤销条目。
`Application.StartNewUndoEntry()` 的作用是创建一个"撤销边界"，之后的操作会成为一个新的撤销条目。

**正确用法**：在每个独立操作之前调用 `StartNewUndoEntry()`，这样每个操作都会成为独立的撤销条目。

**平台支持情况**：

| 平台       | 支持状态 | 说明                                                    |
| ---------- | -------- | ------------------------------------------------------- |
| PowerPoint | ✅ 支持  | 每个操作前调用 `StartNewUndoEntry()` 创建撤销边界       |
| WPS        | ✅ 无需  | WPS 的每个 COM 操作本身就是独立的撤销记录，无需额外处理 |

**实现方式**：

当前实现使用 `CreateUndoScope` 方法，返回 `IDisposable`，通过 `using` 语句自动管理撤销边界：

```csharp
// RibbonCallbacks.cs - 使用 using 语句包裹操作
private void ExecuteAlignment(AlignmentType alignmentType)
{
    try
    {
        var shapes = GetSelectedShapes();
        if (shapes == null || !shapes.Any())
        {
            ShowMessage("请先选择要对齐的形状");
            return;
        }

        var service = UniversalIntegration.GetService<IAlignmentService>();
        if (service == null)
        {
            ShowMessage("对齐服务不可用");
            return;
        }

        // 使用撤销作用域包裹操作，自动处理撤销边界
        using (CreateUndoScope($"对齐: {alignmentType}"))
        {
            service.Align(shapes, alignmentType, _currentReference);
        }

        UniversalIntegration.Logger?.LogInformation($"Alignment executed: {alignmentType}");
    }
    catch (Exception ex)
    {
        UniversalIntegration.Logger?.LogError($"Alignment failed: {ex.Message}", ex);
        ShowMessage($"对齐操作失败: {ex.Message}");
    }
}

// CreateUndoScope 实现
private IDisposable CreateUndoScope(string operationName)
{
    try
    {
        var undoService = UniversalIntegration.GetService<IUndoService>();
        var context = UniversalIntegration.Context;
        if (undoService != null && context != null)
        {
            return undoService.CreateUndoScope(context, operationName);
        }
    }
    catch
    {
        // 撤销作用域创建失败不应影响主操作
    }
    return new NullDisposable(); // 返回空实现，确保 using 语句正常
}
```

**已覆盖的操作**：

- 对齐操作（左/右/上/下/水平居中/垂直居中）
- 分布操作（水平/垂直）
- 尺寸操作（等宽/等高/等大小）
- 三线表格式化
- 毛玻璃卡片创建

**最佳实践**：

- 所有修改操作都应使用 `CreateUndoScope` 包裹
- 操作名称应清晰描述操作内容，便于用户在撤销列表中识别
- 即使撤销服务不可用，也不应影响主操作执行

  - 在 Adapter 层统一捕获 COM 异常，转换为业务异常

- **用户友好的错误提示**：

  - 避免直接显示技术异常信息
  - 提供操作建议（如"请先选择形状"）
  - 记录详细错误到日志文件

- **批量操作的错误恢复**：

  - 批量操作中单个项目失败时，继续处理其他项目
  - 最后汇总显示成功/失败数量

- **实现示例**：
  ```csharp
  public void BatchOperation(IEnumerable<IShapeContext> shapes, IProgressReporter progress)
  {
      int success = 0, failed = 0;
      foreach (var shape in shapes)
      {
          try
          {
              // 操作...
              success++;
          }
          catch (Exception ex)
          {
              _logger.LogError($"处理形状失败: {ex.Message}", ex);
              failed++;
          }
      }
      progress?.Complete($"完成：成功 {success}，失败 {failed}");
  }
  ```

### 5.5 性能优化建议

**现状分析**：

- 当前功能主要面向单次操作，性能影响较小
- 批量操作（如格式化所有表格）可能涉及大量 COM 调用

**优化策略**：

- **减少 COM 调用次数**：

  - 批量读取属性，减少跨进程调用
  - 使用事务性操作（如 PowerPoint 的 `Application.ScreenUpdating = false`）

- **延迟计算**：

  - 仅在需要时访问形状属性
  - 使用缓存机制存储已读取的属性

- **异步操作**（可选）：

  - 对于耗时操作，考虑使用后台线程
  - 注意 COM 线程模型限制（STA）

- **实现建议**：
  ```csharp
  // 在 Adapter 层实现批量操作优化
  public void BatchSetBounds(IEnumerable<IShapeContext> shapes, ShapeRect bounds)
  {
      // PowerPoint: 关闭屏幕更新
      if (_context is PowerPointContext pptContext)
      {
          pptContext.Application.ScreenUpdating = false;
          try
          {
              foreach (var shape in shapes)
              {
                  shape.Bounds = bounds;
              }
          }
          finally
          {
              pptContext.Application.ScreenUpdating = true;
          }
      }
  }
  ```

- 在 Adapter 层封装差异，业务层无需关心

- 提供功能的部分实现，提示用户完整功能仅在 PowerPoint 可用
- 记录日志，便于后续完善

- ✅ 不同语言版本（中文/英文）

### 7.4 已知平台限制

**WPS 平台限制**：

1. **imageMso 图标**：不支持 Office 内置图标，需使用自定义图标
   - 当前实现：在 `GetCustomUI` 中自动移除 `imageMso` 属性
   - 改进建议：实现图标映射机制，自动替换为自定义图标（见 6.4.3 节）
2. **COM 调用方式**：使用 `dynamic` 关键字，性能略低于强类型绑定
3. **API 覆盖度**：部分 PowerPoint API 在 WPS 中可能不存在或行为不同

**处理建议**：

- 在 `WPSHelper` 中统一封装 COM 调用
- 使用 try-catch 捕获不支持的 API 调用
- 在文档中明确标注功能支持情况
